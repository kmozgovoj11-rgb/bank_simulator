# AGENTS.md

## Обзор проекта

Учебный проект "Банковский симулятор".
Проект моделирует базовые банковские сценарии с акцентом на ООП, SOLID, чистую архитектуру и понятное разделение ответственности между слоями.

Текущая версия проекта включает:
- доменную модель клиентов, пользователей, счетов и транзакций;
- прикладные сервисы для регистрации, аутентификации, переводов и истории операций;
- SQLite-хранилище;
- точку входа для локального запуска.

## Текущее состояние проекта

Сейчас проект является backend-ориентированным Java-приложением.
В проекте уже реализованы:
- создание клиентов;
- регистрация и аутентификация пользователей;
- работа со счетами разных типов;
- переводы между счетами;
- хранение истории операций;
- SQLite persistence;
- атомарность перевода через транзакционный брокер.

## Архитектура

В проекте используются слои:
- presentation;
- application;
- domain;
- infrastructure.

Правило зависимостей:
- presentation зависит от application;
- application зависит от domain;
- infrastructure зависит от domain;
- domain не зависит от верхних слоев.

Основной принцип:
- бизнес-правила должны оставаться в domain и application;
- детали хранения данных должны оставаться в infrastructure.

## Доменные правила

При любых изменениях необходимо сохранять следующие инварианты:

- `Customer` и `User` — разные сущности.
- Логин пользователя — телефон.
- Телефон приводится к каноническому формату и валидируется.
- Пароль валидируется до сохранения.
- Пароль не хранится в открытом виде, используется хеш.
- Операции по счетам со статусом `FROZEN` или `CLOSED` запрещены.
- Нельзя закрыть счет с ненулевым балансом.
- Кредитный счет нельзя закрыть при наличии долга.
- Перевод на тот же самый счет запрещен.
- История операций должна сохраняться корректно.
- Rollback-логика транзакций должна сохраняться.
- Read-only представления истории не должны использоваться как исполняемые транзакции.

## Слой application

Application-сервисы отвечают за сценарии использования, а не за хранение данных.
Правила:
- новые пользовательские сценарии добавлять через сервисы приложения;
- не переносить SQL и инфраструктурные детали в application;
- при переводах сохранять атомарность через `TransactionBroker`;
- не обходить сервисы прямыми вызовами репозиториев без необходимости.

## Слой domain

Domain — это ядро проекта.
Правила:
- не добавлять зависимости на SQLite, JDBC или presentation;
- сохранять инкапсуляцию бизнес-правил в сущностях и транзакциях;
- при расширении модели предпочитать полиморфизм и существующую структуру, а не разрастание условных операторов;
- не выносить доменные проверки во внешний код, если они относятся к самим сущностям.

## Слой infrastructure

Infrastructure содержит технические реализации.
Правила:
- SQLite-логика и SQL должны оставаться в инфраструктурном слое;
- репозитории должны реализовывать существующие интерфейсы;
- схема БД должна оставаться согласованной с доменной моделью;
- при изменении persistence-логики не ломать контракты репозиториев.

## Presentation и запуск

Presentation сейчас выполняет роль точки входа и локального запуска проекта.
Правила:
- не размещать здесь бизнес-логику;
- точка входа должна только собирать зависимости и запускать приложение;

## Работа с изменениями

- Сначала изучать существующую реализацию, потом вносить изменения.
- Предпочитать маленькие, локальные и безопасные правки.
- Не трогать несвязанные части проекта.
- Не дублировать существующие модели, репозитории и сервисы.
- Если задача решается расширением текущего кода, не создавать параллельную реализацию.
- Если меняется поведение системы, при необходимости обновлять документацию.

## Проверка результата

### После изменений:
- убедиться, что проект по-прежнему компилируется и запускается;
- проверить затронутый сценарий вручную, если автотестов нет;
- отдельно проверять сценарии регистрации, аутентификации, перевода и истории, если изменение могло их затронуть;
- если проверка не выполнялась, это нужно явно указывать.

### Делать всегда
- сохранять слоистую архитектуру;
- сохранять доменные инварианты;
- следовать существующему стилю именования и организации кода;
- использовать существующие абстракции репозиториев и сервисов.

### Сначала согласовать
- изменение схемы базы данных;
- добавление новой библиотеки;
- крупный рефакторинг через несколько слоев;
- изменение основных доменных контрактов;
- изменение формата хранения или транзакционной модели.

### Не делать
- не ломать архитектурные границы слоев;
- не удалять доменные проверки без причины;
- не переносить SQL в application или domain;
- не заменять рабочую логику временными заглушками.
