# BankSimulator

**Симуляор банка** - учебный проект, который демонстрирует принципы ООП через реализацию полноценного симулятора банка с тремя типами счетов, системой транзакций, начислением процентов и сохранением данных после закрытия программы.

## Основные функции

- **Управление счетами:** Поддержка трёх типов банковских счетов (дебетовый, депозитный, кредитный) с уникальной логикой начисления средств и комиссий.
- **Транзакции и переводы:** Возможность перевода средств между счетами по уникальному номеру счета, генерация истории операций.
- **Аутентификация:** Регистрация и вход по телефону (логин) и паролю.
- **Персистентность данных:** Сохранение состояния счетов и пользователей в базе данных (БД) после закрытия программы, что позволяет восстанавливать сессию при повторном входе.
- **Надёжность и обработка ошибок:** Валидация входных данных, проверка достаточности средств, журналирование действий пользователя и защита от некорректных операций.
- **Расширяемость:** Архитектура позволяет легко добавлять новые типы счетов, комиссий или банковских продуктов без изменения ядра системы.

## Технический стек

- **Язык программирования:** Java 17 (backend), TypeScript (frontend), сборка через Maven (backend) и npm (frontend) — строгая типизация и поддержка ООП.
- **Инструменты для диаграмм:** PlantUML (для построения UML-диаграммы классов и отображения архитектуры проекта).
- **Архитектура:**  Многослойная: веб-API, сервисы, доменная модель, отдельно — хранение в SQLite.
- **Принципы и паттерны:** SOLID, применение порождающих и структурных паттернов для гибкого создания объектов и обработки событий.
- **Хранение данных:** Реляционная БД (SQLite) для сохранения пользователей, счетов и истории операций.
- **Тестирование:** Модульные тесты для проверки core-логики, рефакторинг и код-ревью на всех этапах разработки.

## Принципы SOLID

**S — Single Responsibility (единственная ответственность).** `AuthService` отвечает только за регистрацию и аутентификацию пользователя, `BankService` — за банковские сценарии (клиенты, переводы, история). Разные зоны ответственности не смешаны в одном классе.

**O — Open/Closed (открыт для расширения).** Новый тип счёта добавляется наследником `Account` с собственной логикой (например, `CreditAccount` переопределяет снятие и закрытие), не переписывая заново всю абстракцию «счёт» для остального кода, который работает с типом `Account`.

**L — Liskov Substitution (подстановка подтипов).** `DebitAccount`, `SavingsAccount` и `CreditAccount` используются там, где ожидается `Account` (например, в `TransferTransaction`), при этом не нарушают базовые инварианты (операции для неактивных счетов по-прежнему запрещены на уровне `Account`).

**I — Interface Segregation (разделение интерфейсов).** Доступ к данным разбит на узкие контракты: `AccountRepository`, `CustomerRepository`, `UserRepository`, `TransactionRepository`, отдельно `TransactionBroker` — сервис зависит только от нужных абстракций, а не от одного «репозитория всего».

**D — Dependency Inversion (инверсия зависимостей).** `BankService` зависит от интерфейсов репозиториев и `TransactionBroker`, объявленных в домене; конкретные реализации SQLite подставляются в `Main` при сборке приложения, поэтому application/domain не знают про JDBC напрямую.


## Паттерны проектирования

В проекте явно реализованы три паттерна проектирования `Facade`, `Factory Method` и `Command`.

**Facade**

Паттерн Facade нужен, чтобы предоставить простой интерфейс к сложной подсистеме. В проекте банковская логика включает работу с клиентами, счетами, транзакциями, репозиториями и БД-транзакциями. Чтобы внешний код не работал со всеми этими классами напрямую, используется фасад.
В проекте фасад реализован через интерфейс:
`application/service/BankFacade.java`
И его реализация: 
`application/service/BankService.java`
BankFacade задает простой набор операций:
- создать клиента;
- перевести деньги;
- получить историю счета;
- найти счет.

BankService скрывает детали реализации: обращение к репозиториям, создание TransferTransaction, запуск операции внутри TransactionBroker, сохранение транзакции и обновление счетов. Например, при переводе внешний код вызывает только transferMoney(), а внутри сервис сам находит счета, создает транзакцию, выполняет ее и сохраняет результат.
Таким образом, Facade упрощает использование банковской подсистемы и уменьшает связность внешнего кода с внутренними деталями проекта.

**Factory Method**

Паттерн Factory Method нужен, чтобы вынести создание объектов в отдельное место и не привязывать остальной код к конкретным классам создаваемых объектов.
В проекте этот паттерн реализован в классе: `domain/model/AccountFactory.java`
Фабрика создает конкретный тип счета по значению accountKind:
- `DEBIT` -> `DebitAccount`;
- `SAVINGS` -> `SavingsAccount`;
- `CREDIT` -> `CreditAccount`.

Используется фабрика в классе:
`infrastructure/persistence/sqlite/SqlAccountRepository.java`

Когда счет загружается из базы данных, в таблице хранится строковое поле `account_kind`. Репозиторий читает данные из БД и передает их в `AccountFactory`, а фабрика решает, какой конкретный класс счета нужно создать.

Это удобно, потому что логика создания разных типов счетов собрана в одном месте. Если позже появится новый тип счета, например BusinessAccount, его можно будет добавить в фабрику без размазывания `new BusinessAccount(...)` по разным частям проекта.

**Command**

Паттерн Command нужен, чтобы представить действие как отдельный объект. Такой объект можно выполнить, сохранить, передать, а при необходимости откатить.
В проекте паттерн реализован через интерфейс:
`domain/model/Command.java`
Интерфейс задает два метода:
- execute() - выполнить команду;
- rollback() - откатить команду.
Абстрактный класс:
`domain/model/Transaction.java`
реализует `Command`, а конкретные банковские операции наследуются от него:
- DepositTransaction - пополнение счета;
- WithdrawTransaction - снятие денег;
- TransferTransaction - перевод между счетами.
Каждая транзакция сама знает, как выполнить свою операцию и как ее откатить. Например, TransferTransaction.execute() списывает деньги с одного счета и зачисляет на другой, а rollback() выполняет обратные действия.
Такой подход отделяет сам запрос на выполнение операции от кода, который эту операцию вызывает. В результате банковские операции оформлены как самостоятельные объекты-команды, что делает код понятнее и расширяемее.
